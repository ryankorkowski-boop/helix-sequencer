# Legal Learning Policy

Date: 2026-05-01  
Status: Active

## 1. Mission

Helix builds sequencing intelligence from lawful, generalized methods.  
Helix does not copy protected expression, proprietary sequence structures, or source-specific choreography.

## 2. Core Principle

Use this pattern for all learning and rule extraction:

1. Learn process and method.
2. Rewrite in Helix-original language.
3. Generalize into reusable technical rules.
4. Validate with Helix tests, generated outputs, and checks.

## 3. Allowed Inputs

Allowed sources include:

1. Public official documentation and release notes.
2. Public project repositories and public issue discussions.
3. Public community troubleshooting discussions.
4. User-authored notes and user-owned assets.
5. Helix-generated experiments and benchmarks.
6. Open-license examples when license terms clearly permit the intended use.

## 4. Forbidden Inputs

Do not ingest, mirror, scrape, bulk-download, or preserve:

1. Paid sequence files or paid training materials.
2. Private-group/forum content not publicly accessible.
3. Transcript/caption dumps from video content.
4. Proprietary sequence internals copied from third parties.
5. Timestamped recipe extraction from tutorials.
6. Any material with unclear rights provenance.

## 5. Forbidden Outputs

Do not output:

1. Direct or near-direct choreography reproduction.
2. Timeline-by-timeline imitation of third-party shows.
3. Source-identifying "copy this creator style" instructions.
4. Sequence logic that is traceable to one protected source.

## 6. Safe Transformation Standard

Every Helix rule must be:

1. Original wording.
2. Source-agnostic.
3. General enough to apply across many songs and layouts.
4. Technically testable.
5. Non-derivative in expressive structure.

## 7. Provenance And Traceability

Each research batch should store:

1. Source category (official docs, repo, community, user-owned, internal test).
2. URL or file reference.
3. Extraction date.
4. Confidence level.
5. Whether the final rule is authoritative behavior or community inference.

## 8. Source Trust Hierarchy

Priority order:

1. Official manual and official release notes.
2. Official repository documentation and code behavior.
3. Public community guidance.
4. Internal experiments and local heuristics.

If sources conflict:

1. Prefer authoritative sources.
2. Build a small reproducible test.
3. Record the chosen interpretation and why.

## 9. Pre-Merge Legal Gate

Before merge, confirm:

1. No copied or near-copied text.
2. No proprietary sequence structure retention.
3. No source-specific choreography replication.
4. No private or paid content leakage.
5. All new rules are generalized and testable.

## 10. Incident Response

If questionable material appears:

1. Stop the merge.
2. Identify file path and affected lines.
3. Classify risk: low, medium, high.
4. Remove questionable material.
5. Rewrite as generalized Helix-original policy.
6. Re-run tests and checks.
7. Document whether history cleanup is required.
